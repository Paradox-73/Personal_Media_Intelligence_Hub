## The critical ordering issue: the frozen fold registry

Here's the trap. Your entire evaluation rests on the frozen fold registry — same items in same folds across every experiment, which is what makes your paired tests valid. If you fix data points and regenerate matrices, you have to ask: **does the set of items change?**

- If you're only *correcting fields* on existing items (same `source_id`s), the registry still holds — same items, same folds, just better feature values. Safe.
- If fixing the data *adds or removes* items (e.g., a mis-fetched item gets dropped, or a corrected one gets a new ID), the registry is now stale — it references items that changed identity, and your "frozen" comparison is silently broken.

So instruction one to the agent: **after fixing data and before re-running anything, validate that every `source_id` in the registry still exists and still maps to the same item.** If the item set changed, the registry must be regenerated — and if you regenerate the registry, *every historical number in the paper was computed on different folds and is no longer strictly comparable.* That's not necessarily fatal (you re-run everything on the new registry), but you need to know it happened rather than discover it as a mystery discrepancy. The agent has a habit of catching these; make it check explicitly.
