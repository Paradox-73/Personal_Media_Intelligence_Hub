## PHASE 1 — Gatekeeper experiments (2–3 days)

- On registry folds: local SVR **with** vs **without** the unified-prior feature. Metrics: MAE, skill score. Paired Wilcoxon per domain.
- **Decision rule:** prior not significantly helpful (p ≥ 0.05 or negative effect) → remove it from that domain, record the null. This decides whether the unified model has any production role.

## PHASE 2 — THE TRANSFER GRID (the centerpiece, ~1 week)

### Task 2.1 — Build `src/experiments/transfer_study.py`
One harness, config-driven:
- **Targets:** movies, tv, games, books. **Sources:** all 15 non-empty subsets of the other three domains, each run with and without music (pseudo-labeled, sample_weight 0.3–0.5) → 30 source configs per target.
- **Features:** shared-space only (aligned vibe PCA, unified genre encoding, critic-average, year, log-popularity) + missingness masks. Domain-specific blocks excluded. CORAL fit on training folds only.
- **Protocols per (S, T) cell:**
  - **A (baseline):** train on T only — registry folds. Already have these numbers; load, don't recompute.
  - **B (zero-shot):** train on S, evaluate on all of T's registry test folds.
  - **C (augmented):** train on S + {0%, 25%, 50%, 100%} of T's training folds, evaluate on T's test folds. Target-fraction subsampling must be deterministic (seed from registry).
- **Model:** the frozen Mean Ensemble (XGB+Cat), default unified hyperparameters. No per-cell tuning — that would be 120 tuning runs of overfit.
- **Metrics per cell:** MAE, skill score (vs T's training-fold mean), Spearman ρ, paired Wilcoxon vs Protocol A on identical test items.
- **Output:** `reports/transfer_grid_results.csv` (one row per cell × protocol × fraction × repeat) + `transfer_grid_summary.json` for the renderer.
- **Accept when:** the full grid runs end-to-end from one command and is reproducible (rerun → identical numbers).

### Task 2.2 — Analysis & verdict
- **Affinity heatmap:** zero-shot skill score, single-source → target (4×4 + music row).
- **Best-subset table:** per target, the source combo with highest significant augmented lift at the 100% fraction; mark "none" where nothing beats Protocol A.
- **Learning curves:** per target, skill vs target-fraction for (best S) vs (T alone). The crossover point — if any — is the headline figure.
- **Similarity↔transfer correlation:** compute pairwise domain MMD + centroid distance in the aligned space; Spearman-correlate with zero-shot skill. This is the "relations between domains" deliverable.
- **Decision rules (pre-registered, write them in the report BEFORE looking at results):**
  - Cell is a *positive transfer finding* iff augmented lift > 0, p < 0.05 paired, at ≥2 target fractions.
  - Project verdict "taste transfers" iff ≥1 target has a positive finding; otherwise verdict is the null: "in the shared feature space, no domain combination beats local models — transfer must be item-level."
- **Accept when:** heatmap + curves render in a new Streamlit page (`15_Transfer_Atlas.py`) and the verdict paragraph (whichever it is) is in both docs.

The per-domain slice finally landed and it's the most important table the project has produced — but this cycle also quietly changed what "Unified" means, and that needs to be unwound before any number from it can be trusted. In order of importance:

## 1. The Unified headline switched populations mid-comparison

Last cycle: Unified = 1,264 rated items, R² 0.444. This cycle: Unified = 4,952 items, R² 0.503 — because 3,688 music tracks with **PU pseudo-labels** entered the evaluation pool. That 0.444 → 0.503 "improvement" is not a model improvement; it's a population swap. Worse, 74% of the eval set now consists of the model predicting *another model's output* — the PU classifier's calibrated affinities — and if any PU-derived affinity features sit in the input vector, there's partial circularity: the model is being graded on reconstructing a label its own features encode. The 0.281 music R² suggests the circularity isn't total, but the pooled headline is now a blend of "predicting your actual ratings" and "predicting a pseudo-label," weighted 1:3 toward the latter.

Fix: report **two unified rows, permanently** — `Unified (rated, N=1,264)` as the headline for your actual-taste claim, and `Unified (full pool incl. PU pseudo-labels, N=4,952)` as a secondary line. The README footnote gestures at this but the benchmark table presents 0.503 as *the* number, which a careful reviewer will correctly read as metric inflation — the exact sin your case study brags about avoiding. Also: the same swap silently invalidated the ablation comparison, see §3.

## 2. The per-domain slice is a verdict — read it as one

Put the slice next to the standalone models:

Movies 0.549 unified vs 0.612 standalone. Shows 0.173 vs 0.325. Games MAE **1.097** unified vs 0.773 standalone. Books 0.012/0.611 vs −0.002/0.576. 

**The unified model is worse than the local model in every single domain.** Pooled 0.503 is held up entirely by movie volume and music pseudo-labels. This is the "movies + passengers" outcome I flagged as the thing the slice would test, and it tested positive. Three consequences:

- The unified model's *only* legitimate production role right now is as the **distillation prior** for Games/Books — and even that is now suspect, because a prior whose own Games MAE is 1.097 may be anchoring the local SVR badly. Run the missing ablation: Games/Books local SVR *with* vs *without* the unified-prior feature, paired Wilcoxon on the frozen folds. If the prior isn't significantly helping, the unified model currently helps nothing, and that's essential to know before the grid.
- This is not a project failure — it's the precise empirical motivation for the transfer grid. The question was never "does one pooled model beat local models" (answer: no); it's "which *source combinations* transfer into which targets, zero-shot and augmented." The slice proves naive pooling isn't the mechanism. The grid tests whether any mechanism exists.
- For the docs: this finding deserves its own subsection, stated plainly. "Naive cross-domain pooling underperforms local models in all four domains" is an honest, interesting result — burying it under a pooled 0.503 headline reverses the story.

## 3. The ablation report regressed from 4 rows to 2 — and its baseline teleported

Keep the mean ensemble (it's free), but report the effect size alongside p, and don't let the report imply the ensemble matters. The simplex converging to [0.45, 0.55] ≈ equal weights is itself the evidence that neither member dominates.

## 1. The renderer is publishing unified slices as standalone domain models — with 5× inflated N

Look at the N column: Movies 4,900, TV 795, Games 310, Books 315. Those are exactly **980×5, 159×5, 62×5, 63×5** — item count × 5 CV repeats. The writer is counting OOF prediction *rows* (one per item per repeat) as N. And the smoking gun is that the "benchmarks" array and the "slices" array in `latest_metrics.json` are *identical, row for row*: Movie benchmark MAE 0.48387755… = movie slice MAE 0.48387755…. The per-domain benchmark rows aren't the standalone models at all — they're the **unified model's per-domain OOF slice**, with the old model labels ("CatBoost-Stack", "Simplex-Stack", "Distilled SVR") glued on top.

Consequences in the published docs right now: README claims Movies R² 0.541 — your real standalone Movies model is 0.612. TV shows 0.167 instead of 0.325. Games MAE 1.116 instead of 0.773. Every standalone number got silently replaced by the (weaker, as we established) unified slice, while keeping the standalone models' names. This is precisely the failure the renderer was built to prevent, except now it's one bug in one script instead of scattered hand-edits — which is the system working, in a way: it's findable and fixable in one place.

Fix: the writer must pull standalone metrics from each domain pipeline's own OOF output, deduplicate to per-item before computing N (and aggregate the 5 repeats per item — mean prediction or mean error, pick one and document it), and keep `slices` as a separate section explicitly labeled "unified model, by domain." Acceptance check to add: `assert benchmarks != slices` and `assert N == n_unique_items`.
